# diff

## Problem Statement

`cutting-garden capture` writes a content-addressable receipt
recording every entry of a captured tree, and
`cutting-garden restore` materializes a tree from a receipt
([FDR 0001]). Together they form a closed round-trip, but neither
answers the natural follow-up: **is this on-disk tree what the
receipt says it should be?**

Today the only way to answer that question is to restore into a
fresh location and shell out to `diff -r` between the two trees.
That doubles disk usage, requires a writable destination, and
obscures the per-entry shape of the differences (mode vs content
vs type) behind text-diff noise. Receipt verification is the
canonical use case — confirming a backup is intact, detecting
config drift, gating a deploy on a known-good tree — and none of
those callers want a second materialization.

`cutting-garden diff <receipt-id> <dir>` fills the gap. It walks
`<dir>` and the receipt's entries in lockstep, recomputing each
on-disk file's content blob-id under the receipt's source-store
hash family **without persisting anything**, and reports per-entry
differences. Same store-hint resolution as restore; same path
sanitization as restore; no second materialization.

## Interface

### Synopsis

    cutting-garden diff [-store <id>] [-verify-blobs-exist] <receipt-id> <dir>

### Positional arguments

- `<receipt-id>` — the markl-id of a
  `cutting_garden-capture_receipt-fs-v1` blob. Required.
- `<dir>` — the directory whose contents are compared against the
  receipt. MUST exist and MUST be a directory. Required.

### Flags

- `-store <id>` — explicit blob-store-id to resolve `<receipt-id>`
  against. Overrides the receipt's store-hint resolution.
  Inherits exactly the semantics from `restore -store`
  ([FDR 0001 §Store-Hint Resolution]). When omitted, the consumer
  follows the same hint-driven flow.
- `-verify-blobs-exist` — turns on the receipt-vs-store check on
  top of the default tree-vs-receipt comparison. See
  [Receipt-vs-store probe](#receipt-vs-store-probe). Off by
  default because it adds one `HasBlob` round-trip per file entry,
  which is meaningful on remote (e.g. SFTP) stores.

### Exit codes

- `0` — `<dir>` matches the receipt; no differences emitted.
- nonzero — at least one difference was emitted, or an error
  prevented the comparison. The two are distinguishable from
  stderr: a clean diff prints `diff: N differences`; an error
  prints the framework's error tree.

The diff(1)-style 0/1/2 split (mismatch vs error) was considered
and explicitly deferred — see [Limitations](#limitations).

### Behavior

The comparison runs in five phases:

1. **Validate `<dir>`.** Stat it. If it does not exist, refuse
   with a `directory does not exist` diagnostic. If it exists but
   is not a directory, refuse with a `not a directory` diagnostic.
   No store reads, no blob fetches.

2. **Resolve the source store + parse the receipt.** Use `-store`
   if set, else fall back across configured stores via `HasBlob`
   probes (same logic as restore phase 1). Parse the blob via
   `capture_receipt.Coder.DecodeFrom`. The result is a `*V1`
   carrying optional `Hint` and `Entries`.

3. **Sanitize the receipt's entries.** Per [RFC 0003] §Path
   Sanitization, refuse the entire comparison — before any walk —
   if any entry has an empty root, a NUL byte in root or path, or
   would materialize outside `<dir>`. The check is
   identical to restore's, performed by the shared
   `validateEntries` helper.

4. **Resolve the effective source store via hint.** Per [RFC 0003]
   §Store-Hint Resolution. Diff inherits restore's branches
   verbatim: `-store` wins, then hint match, then config-drift
   refusal, then fallback-with-notice. The hint resolution
   produces the `FormatHash` used in step 5.

5. **Walk `<dir>` and compare.** Walk via `filepath.WalkDir`
   (lstat-based, no symlink follow — same rule as capture's
   `walkRoot`). For each on-disk regular file, stream its content
   through a **discard blob store**
   (`foxtrot/blob_stores.NewDiscardBlobStore`) under the resolved
   hash family. The discard store is a virtual `BlobStore` whose
   `MakeBlobWriter` builds `blob_io.NewWriter(...)` over
   `io.Discard`: the digester chain still sees every byte (so the
   resulting markl-id matches what a real store would produce),
   but no bytes are persisted. The disk's per-entry tuple is then
   compared against the receipt's tuple at the same materialized
   path, and per-attribute differences are emitted.

6. **(Optional) Probe the source store for missing blobs.** When
   `-verify-blobs-exist` is set, the consumer also probes the
   resolved source store via `HasBlob` for every receipt entry
   with a non-empty `BlobId` (i.e. every file entry). Misses
   produce `B` lines that count toward the differences total.
   See [Receipt-vs-store probe](#receipt-vs-store-probe).

#### Destination preconditions

Inverse of [FDR 0001 §Destination Preconditions]:

- `<dir>` MUST exist at invocation time.
- `os.Lstat(dir)` is checked before any other work. Failure cases:
    - Path does not exist:

          error: <dir>: directory does not exist
          hint: pass an existing directory; diff does not materialize anything

    - Path exists but is not a directory:

          error: <dir>: not a directory (mode <mode>)

  Exit nonzero in both cases.

#### Sanitization

Identical to [FDR 0001 §Sanitization]. A receipt that fails
sanitization (NUL bytes, empty root, escape) is refused before
the walk; the receipt is the same artifact restore consumes, so
the same rules apply.

The diagnostic format mirrors restore's: a single `error:`
summary followed by indented `key: value` payload lines. Diff
does not introduce new diagnostic shapes.

#### Source-store resolution

Identical to [FDR 0001 §Store-Hint Resolution]. Diff differs from
restore only in what it does with the resolved store: where
restore reads blobs to materialize them, diff uses only the
resolved store's `GetDefaultHashType()` to select the hash family
for the discard store. Diff itself never reads blobs from the
source store.

A consequence: diff's `-store <id>` does not have to point at a
store that holds the receipt's referenced blobs. It only has to
point at a store whose default hash family matches the family
the blob-ids were computed under. This is symmetric with how
restore uses the source store, but in practice diff's contract
is looser since it doesn't fetch.

#### Comparison rules

The comparison key for both sides is the rel-to-`<dir>`
materialization path:

- For a receipt entry `e`:
  `filepath.ToSlash(filepath.Clean(filepath.Join(e.Root, e.Path)))`
- For a disk entry walked under `<dir>`:
  `filepath.ToSlash(filepath.Clean(rel))` where
  `rel = filepath.Rel(dir, p)`.

When the receipt was produced by a single-root capture, `e.Root`
is `"."` ([RFC 0003 §Root Encoding]) and the receipt key collapses
to `e.Path`. This matches the disk-walk's rel-to-`<dir>` key
directly when `<dir>` is the originally-captured directory, so
`cg capture src; cg diff <rid> src` reports clean without an
intermediate `restore`. Multi-root receipts keep distinct `e.Root`
values per top-level dir, so the capture-symmetric `<dir>` is the
capture-time PWD that contained all roots — there is no symmetric
form for diffing one of several roots against a multi-root receipt.

Walking the union of keys yields four cases:

1. Key in receipt only → emit `D <key>\t<type>` (Deleted).
2. Key on disk only → emit `A <key>\t<type>` (Added).
3. Key in both, types differ →
   emit `T <key>\t<recv-type> -> <disk-type>` (Type-changed).
4. Key in both, same type → run the per-type diff (see below).

The on-disk `.` entry — the `<dir>` argument itself — is **always
filtered out from the disk side unless the receipt has a
matching `.` key** (i.e. the receipt was captured with `Root="."`
and `Path="."`). The dir is the container the receipt's tree
materializes into, not part of the receipt's contents; reporting
`A  .` for every default-positioned diff would be noise.

Output is sorted lexicographically by key, so test assertions and
visual scans can rely on ordering.

#### Per-type comparison

For matched entries with the same type:

- **`file`.** Compare `Mode.Perm()`; emit `M <key>\tmode <a> -> <b>`
  on mismatch. Compare `BlobId`; emit
  `M <key>\tblob <a> -> <b>` on mismatch. Mode and blob are
  reported as separate lines so a single path can produce
  multiple diff entries.

- **`dir`.** Compare `Mode.Perm()`; emit
  `M <key>\tmode <a> -> <b>` on mismatch.

- **`symlink`.** Compare `Target` (the literal symlink target
  string, not the resolved path). Emit
  `M <key>\ttarget "<a>" -> "<b>"` on mismatch. Mode is not
  compared — symlinks have system-dependent mode handling and
  capture does not normalize it.

- **`other`.** No comparison; capture is lossy here ([FDR 0001
  §Limitations]) — the receipt records `other` with no
  comparable content, and the disk walk likewise can't classify
  it. Skipped silently.

#### Receipt-vs-store probe

`-verify-blobs-exist` adds an orthogonal check on top of the
default tree-vs-receipt comparison. It catches a failure mode
the default comparison cannot: a receipt that references blobs
no longer present in the source store (gc'd, hand-crafted with
a bogus id, partially synced from a remote).

When the flag is set, the consumer iterates `v1.Entries`. For
every entry where `Type == file` and `BlobId != ""`, the
recorded id is parsed and probed via the resolved source
store's `HasBlob`. Two outcomes produce a `B` line:

- `BlobId` parses but the store reports `HasBlob == false`.
- `BlobId` fails to parse — treated as missing because there
  is no resolvable address.

The line shape is:

    B  <key>\tblob <id> missing in source store

`B` lines are emitted in addition to any `M`/`A`/`D`/`T` line
the same key produces; they describe orthogonal failures. A
single path can produce e.g. both a `D` (path absent on disk)
and a `B` (referenced blob also missing in store), telling the
operator the receipt is doubly broken at that path.

The check is opt-in because it adds one `HasBlob` round-trip
per file entry. On a local hash-bucketed store this is a
`files.Exists` (cheap); on SFTP or other remote backends it is
a stat-over-the-wire and can dominate the wall-clock cost of
diff.

A consequence of the probe: a clean `cutting-garden diff` exit
without `-verify-blobs-exist` only proves the on-disk tree
matches the receipt's *records*, not that restoring from the
receipt would succeed. With the flag set, exit 0 additionally
proves every referenced blob is currently retrievable from the
source store.

#### Output format

Each difference is a single line on stdout:

    M  path/to/file       blob abc1234 -> def5678
    M  path/to/file       mode 0755 -> 0644
    A  path/to/extra      file
    D  path/to/missing    file
    T  path/to/changed    file -> symlink
    B  path/to/file       blob abc1234 missing in source store

Marker is two characters (letter + space) for visual alignment.
Path and detail are tab-separated. After all diff lines, on
mismatch, stderr carries `diff: N differences` and the framework
emits its standard error-tree. JSON / TAP output formats are
deferred (see [Limitations](#limitations)).

## Examples

**Symmetric round-trip** — a restored tree faithfully matches
the receipt:

    $ rid=$(cutting-garden capture src | jq -r '.id')
    $ cutting-garden restore "$rid" out
    $ cutting-garden diff "$rid" out
    $ echo $?
    0

**Detect content drift:**

    $ rid=$(cutting-garden capture src | jq -r '.id')
    $ cutting-garden restore "$rid" out
    $ echo "tampered" > out/src/note.txt
    $ cutting-garden diff "$rid" out
    M  src/note.txt	blob blake2b256-abc... -> blake2b256-def...
    diff: 1 differences
    $ echo $?
    1

**Detect added file:**

    $ echo "new" > out/extra.txt
    $ cutting-garden diff "$rid" out
    A  extra.txt	file
    diff: 1 differences

**Detect type change:**

    $ rm out/src/file && ln -s other out/src/file
    $ cutting-garden diff "$rid" out
    T  src/file	file -> symlink
    diff: 1 differences

**Override store-hint:**

    $ cutting-garden diff -store .work "$rid" out

**Detect missing blobs in the source store** (e.g. after a gc):

    $ cutting-garden diff -verify-blobs-exist "$rid" out
    B  src/note.txt	blob blake2b256-abc... missing in source store
    diff: 1 differences
    $ echo $?
    1

## Limitations

- **Single text output format on first cut.** No `-format
  json|tap` like capture has. JSON output for scripted callers
  (e.g. CI gates) is deferred to a follow-up; structured-output
  consumers can grep the marker prefix today.

- **No diff(1)-style 0/1/2 exit-code split.** Both differences
  and errors map to exit 1. The split would require plumbing a
  custom exit-code path through `cli_main.Run` (which today
  always returns 0 or 1) and a typed error in futility. The
  user-facing distinction lives on stderr instead: a clean
  mismatch prints `diff: N differences`; an error prints
  `cutting-garden: <error tree>`. This may be revisited if
  shell-script callers find the dichotomy load-bearing.

- **`-verify-blobs-exist` is opt-in.** A diff without the flag
  only verifies the on-disk tree against the receipt's records;
  it does not probe the source store for the receipt's blobs.
  A clean exit therefore does not, by itself, prove the receipt
  is restorable. Pass `-verify-blobs-exist` for that stronger
  guarantee. Off by default because the probe adds one
  `HasBlob` round-trip per file entry, which is meaningful on
  remote (e.g. SFTP) stores.

- **No re-hash on `-verify-blobs-exist`.** The probe checks
  blob *presence* via `HasBlob`, not content integrity — a
  blob that exists but has bitrotted in the store will still
  pass. Bitrot detection is the natural follow-up; it would
  layer on top of the same flag (or a `-verify-blobs-deep`
  sibling) as a `B  ... bitrot` line. Today's contract is that
  the source store has not been corrupted out-of-band.

- **No "ignored" patterns.** Diff has no equivalent of
  `.gitignore`; every entry under `<dir>` is compared. Users who
  want to scope a diff to a subset can capture+restore a
  pre-trimmed tree.

- **`type=other` entries are silently skipped.** Receipt records
  them with no comparable content; the disk walk likewise can't
  classify a non-regular, non-symlink, non-dir thing as anything
  but `other`. A future verifier-rules RFC may pin this more
  tightly.

## More Information

- [FDR 0001] specifies the restore consumer's contract. Diff
  reuses restore's path-sanitization and store-hint helpers
  unmodified.
- [RFC 0003] specifies the receipt wire format and the
  capture/restore consumer rules. Diff is a *new* consumer mode
  that does not modify the wire format. If diff's comparison
  semantics need normative pinning across implementations, a
  future RFC patch should add a §Verifier Rules section to
  RFC 0003 rather than introducing a new RFC.
- The discard blob store
  (`go/internal/foxtrot/blob_stores/store_discard.go`) is the
  reusable seam — any future tool that needs to compute a
  blob-id without persisting can construct one with
  `NewDiscardBlobStore(hashFormat)`.
